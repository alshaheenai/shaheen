# Phase 0 — Research & Decisions: Real Publishing (Phase 7)

All "NEEDS CLARIFICATION" from Technical Context are resolved below. Each item: Decision · Rationale · Alternatives considered.

## 1. Email provider & send strategy

- **Decision**: Use the official **`resend` SDK**. Fetch active subscribers, send via **batch endpoint in chunks of ≤100** (`resend.batch.send`), one personalized message per recipient (each needs its own unsubscribe link). Attach an **`Idempotency-Key`** per (issue, batch) so a re-trigger or retry never double-sends. Set **`List-Unsubscribe`** and **`List-Unsubscribe-Post: List-Unsubscribe=One-Click`** headers pointing at the unsubscribe route.
- **Rationale**: Resend keys are already provisioned in env; the SDK gives native batch + idempotency + header support with minimal code. Batching keeps the run fast and within provider limits. One-click List-Unsubscribe is an inbox-provider requirement for bulk senders and protects deliverability/reputation.
- **Alternatives**: (a) Resend **Audiences/Broadcasts** — moves the subscriber list into Resend; rejected because our `subscribers` table + RLS is the source of truth and we need per-send control and our own double opt-in. (b) **Raw fetch** to the Resend REST API (matching the Telegram pattern) — viable and dep-free, but the SDK's batch + idempotency ergonomics justify one well-scoped dependency. (c) `@react-email` templates — rejected; a plain RTL HTML function is simpler and avoids a build-time dep.

## 2. Sending-domain verification (DNS)

- **Decision**: Verify **`alshaheenai.com`** for Resend before enabling the email channel: add the domain in Resend → it returns the required **SPF (TXT), DKIM (CNAME/TXT), and DMARC (TXT)** records → add those records to Cloudflare DNS via the **Cloudflare MCP** → confirm verification in Resend. The `from` address is `news@alshaheenai.com` (env `RESEND_FROM_EMAIL`).
- **Rationale**: Bulk email from an unverified domain is rejected/spam-filed. The exact record values are **generated by Resend per domain** (region-specific), so they can't be hardcoded ahead of time — they're captured during the setup task. DMARC is needed for the one-click unsubscribe / bulk-sender requirements.
- **Alternatives**: Resend's shared `onboarding@resend.dev` sender — rejected (not the brand, poor deliverability, capped). Manual DNS in the registrar dashboard — rejected; Cloudflare manages DNS and the MCP is connected.

## 3. Distribution state model

- **Decision**: **No new column** on `published_issues`. Distribution state is the existing **`channel_results` jsonb**, shaped as `{ "<channel>": { "status": "success|failed|skipped", "at": ISO8601, "summary": {…} } }`. "Ready" = a `published_issues` row exists (created on Editor approval) with that channel absent/not-success. "Published on a channel" = `channel_results.<channel>.status = 'success'`. The **public blog shows only issues where `channel_results.blog.status = 'success'`**.
- **Rationale**: `channel_results` already exists and is the natural single source of truth for per-channel status, idempotency, and the oversight UI. Avoids a migration on `published_issues` and any dual-write/consistency risk. Satisfies FR-003/004/005 directly.
- **Alternatives**: Add `distributed_at`/`status` columns — rejected as redundant with `channel_results` (violates simplicity). A separate `channel_results` table — rejected; the spec's per-channel needs fit a keyed map, and the column already exists.

## 4. Idempotency & retry

- **Decision**: The orchestrator skips any channel already at `status='success'` unless `force=true`. Email additionally uses Resend `Idempotency-Key` per (issue, batch) as a second guard against partial-batch retries. Telegram channel idempotency = the `channel_results.telegram.success` check (no native idempotency on `sendMessage`). Blog is naturally idempotent (render-from-DB).
- **Rationale**: Two-layer guard (our state + provider key) makes re-triggers and crash-recovery safe (SC-003). Per-channel granularity enables "re-trigger only the failed channel" (FR-005).
- **Alternatives**: Storing sent-message ids per recipient — overkill for current scale; the batch idempotency key suffices.

## 5. Double opt-in subscribe flow

- **Decision**: `subscribers` migration adds `confirmation_token text` and `unsubscribe_token text` (both default `gen_random_uuid()::text`) and flips the column default `status` to `'pending'`. Subscribe route inserts/updates a `pending` row and emails a tokenized confirmation link; confirm route sets `status='active'`, stamps `confirmed_at`, clears the confirmation token. Unsubscribe route matches `unsubscribe_token` and sets `status='unsubscribed'`. All three run server-side (admin client). Existing rows keep `status='active'`.
- **Rationale**: Double opt-in is standard anti-abuse + deliverability practice and was the clarified scope. Two stable tokens separate concerns (one-time confirm vs durable unsubscribe). Server-side routes avoid exposing the table to anon writes beyond the existing narrow insert policy.
- **Alternatives**: Single opt-in — rejected (clarified out). Signed stateless tokens (HMAC, no columns) — viable but adds a secret-management/cryptography surface; stored random tokens are simpler and auditable.

## 6. Telegram channel posting (vs admin review bot)

- **Decision**: Reuse the **raw `fetch` `sendMessage`** pattern from `lib/telegram.ts`, posting to the **public channel id** stored in `publishing_channels.config` for channel `telegram` (e.g. `{ "channel_id": "@AlShaheenAi" }`). Content = headline + short summary (TL;DR) + link to the blog page (full read on the blog). Honors Telegram message length by summarizing, never mid-word truncation.
- **Rationale**: The bot + token already exist; only the target differs (public channel vs admin chat). Storing the channel id in `publishing_channels.config` keeps it data-driven (Principle II). A summary+link avoids Telegram's 4096-char limits and drives traffic to the blog.
- **Alternatives**: `grammy` dependency — rejected; raw fetch already works in the repo. Posting the full issue — rejected (length limits, worse UX than the blog).

## 7. Reader-facing rendering (blog + email)

- **Decision**: A shared `IssueBody` renderer (`components/issue-view.tsx`) produces the structured Arabic layout (TL;DR → main + «بعين الشاهين» → roundup → tools) reusing the root layout's RTL + Thmanyah Sans. The blog page is a Server Component reading `published_issues` by slug; `generateMetadata` provides title/description for link previews (FR-016). The email template (`lib/email/templates.ts`) emits **inline-styled RTL HTML** of the same content (email clients ignore external CSS), plus header/footer with brand and unsubscribe.
- **Rationale**: One conceptual layout, two render targets. Inline styles are mandatory for email; the web page uses the existing Tailwind/brand styling. Server Components keep reader pages fast and SEO/share-friendly.
- **Alternatives**: MJML/`@react-email` — rejected for dep simplicity; the layout is simple enough for a hand-written template.

## 8. Absolute base URL for links

- **Decision**: Add **`NEXT_PUBLIC_SITE_URL`** (e.g. `https://alshaheenai.com`, `http://localhost:3210` locally) and a `lib/site.ts` `getBaseUrl()`/`issueUrl(slug)` helper. Email and Telegram links use it; falls back to `brand_config.settings.domain` if the env var is unset.
- **Rationale**: Email/Telegram need absolute URLs; relative paths don't work off-site. An env var is explicit and environment-correct; the brand domain is a sane fallback.
- **Alternatives**: Deriving from request headers — unavailable in background jobs (no request). Hardcoding the domain — rejected (breaks local dev, not data-driven).

## 9. Out-of-scope confirmations

- **X (Twitter)**: deferred (paid API tier ~$100+/mo) — orchestrator stays channel-generic so adding it later = a new `lib/publish/x.ts` + enabling the `x` channel row. **WhatsApp**: not in Phase 7 scope (documented env exists but unused). **Cron scheduling** of the publish trigger: Phase 10 (the trigger is built and manually invokable now).
